## Documentation Directory Structure with Titles

- **About** — overview of NeMo Run’s purpose, concepts, and architecture
  - **About NeMo Run** — what NeMo Run is, the ML experiment lifecycle it supports, and who it’s for.
  - **Why NeMo Run** — problems it solves (manual configs, launchers, tracking), and its roles across configuration, execution, and management.
  - **Key Features** — type‑safe configuration, multi‑environment execution, orchestration, tracking, packaging, and ecosystem integrations.
  - **Core Architecture** — three layers (configuration, execution, management), key components, and data flow with a clickable diagram.
  - **NeMo Run and the NeMo Ecosystem** — how Run orchestrates NeMo libraries from modeling through orchestration to runtime environments.

- **Get Started** — onboarding, install, and first run
  - **Get Started with NeMo Run** — section landing with requirements, install paths, quick links, and suggested next steps.
  - **Install NeMo Run** — install from GitHub/pip, optional extras (Ray, SkyPilot), system deps (Docker/Slurm), and verification.
  - **Quick Start** — minimal path from install to first experiment: define a function, configure with `run.Config`, pick an executor, run.

- **Learning Resources** — tutorials, runnable examples, applied use cases
  - **About NeMo Run Learning Resources** — landing that explains the learning path: tutorials → examples → use cases.
  - **Tutorials** — step‑by‑step guides for core workflows.
    - **Configure Your First Model** — `run.Config` basics, dataclasses, validation, dependent configs, and best practices.
    - **Your First NeMo Run Experiment** — create a training function, configure it, run once and as a small experiment.
    - **Run Your First Experiment** — use `Experiment` and `Partial` to organize multiple runs with meaningful names.
    - **Save and Load Models** — persistence patterns, checkpoints, manifests, and simple registries.
    - **Perform Hyperparameter Tuning** — grid/random/Bayesian tuning, analysis, and visualization utilities.
  - **Examples** — complete, executable patterns you can adapt.
    - **PyTorch Training with NeMo Run** — datasets, models, optimizers, DDP, env selection, saving artifacts.
    - **TensorFlow Training with NeMo Run** — Keras models, distributed strategies, logging, saving, deployment.
    - **LLM Fine‑tuning with NeMo Run** — dataset prep, tokenization, LoRA/QLoRA, Trainer, evaluation, deployment.
  - **Use Cases** — practical workflows by domain.
    - **Collaboration**
      - **Experiment Tracking** — centralized metadata, sharing, dashboards, and team reporting.
    - **Production**
      - **ML Pipelines** — orchestration, dependencies, retries, monitoring, and alerting.
      - **Model Deployment** — versioning, canary/A‑B, scaling, and runtime monitoring.
    - **Research**
      - **Hyperparameter Optimization** — Optuna/MOBO patterns, distributed sweeps, analysis.
      - **Reproducible Research** — full state capture, environment spec, config versioning, snapshots.

- **Guides** — deeper how‑tos for production usage
  - **About NeMo Run Guides** — landing for configuration, execution, management, packaging, Ray, and troubleshooting.
  - **Configuration Guide** — advanced patterns, factories, composition, validation, serialization, CLI integration.
  - **Execute NeMo Run** — executors/backends, packagers, launchers (torchrun/FT), multi‑environment runs, resource/fault tolerance.
  - **Manage NeMo Run Experiments** — tasks, dependencies, logs, artifacts, snapshots, retries, and status monitoring.
  - **Package Code for Deployment** — Base/GitArchive/Pattern/Hybrid packagers, support matrix, examples, best practices.
  - **Deploy Ray Clusters and Jobs** — RayCluster vs RayJob, KubeRay/Slurm configs, workflows, and operational tips.
  - **Troubleshoot NeMo Run** — install/config/execution diagnostics, common errors, and structured debug strategies.

- **Integrations** — frameworks, platforms, monitoring, automation
  - **About NeMo Run Integrations** — landing for frameworks, cloud, monitoring, and CI/CD guides.
  - **ML Frameworks Integration** — PyTorch/TF/JAX/SKLearn/XGBoost/HF patterns: datasets, optimizers, DDP/TPU usage.
  - **Cloud Platforms Integration** — AWS/GCP/Azure/Kubernetes/DGX: executors, credentials, autoscaling, cost controls.
  - **Monitoring Tools Integration** — WandB/MLflow/TensorBoard/Neptune/Comet: metrics, artifacts, unified logging patterns.
  - **CI/CD Pipelines Integration** — GitHub Actions/GitLab/Jenkins/CircleCI/Azure DevOps with sample workflows and artifacts.

- **References** — definitive look‑ups and APIs
  - **About NeMo Run References** — landing with when to use each reference and quick links.
  - **CLI Reference** — entrypoints/factories, flags, export (YAML/JSON), detached/tail logs, executor overrides and examples.
  - **Configuration Reference** — types, validation rules, serialization formats, Fiddle integration, templates and composition.
  - **FAQs** — curated answers and code snippets across install, configuration, execution, and management.
  - **API Reference** — generated per‑module docs (apidocs) for classes, functions, and modules.

## Suggested Reviewer Checklist

- [ ] Information architecture and navigation
  - Top-level sections and order match `docs/index.md`; depth ≤ 3.
  - Sidebars and breadcrumbs mirror the hierarchy for NeMo Run topics.
  - From `apidocs/index` and `references/cli-reference.md`, key API pages (for example, `apidocs/core/core.execution.launcher.md`, `apidocs/cli/cli.md`) are reachable within two clicks from relevant landings.

- [ ] Content placement and duplication
  - Documents live in the correct sections.
  - Guides: configuration (`guides/configuration.md`), execution/launchers (`guides/execution.md`), management (`guides/management.md`), packaging (`guides/packaging.md`), Ray (`guides/ray.md`).
  - Learning Resources: tutorials → examples → use cases form a runnable learning path for NeMo Run workflows.
  - No duplicated substantive content; keep one canonical page and cross-link from others.

- [ ] Onboarding and landing pages
  - Get Started covers install + first run and points to deeper docs.
  - Each landing page states purpose, audience, prerequisites, and clear next steps.

- [ ] Consistency with code and APIs
  - Terminology matches code/CLI and parameter names (for example, `Launcher`, `Runner`, `Packager`, `devspace`, `Experiment`, `FaultTolerance`, `torchrun`, `RayCluster`, `RayJob`).
  - API coverage includes core areas (`apidocs/core/core.execution.*.md`, `apidocs/run/run.*.md`, `apidocs/cli/cli.md`, `apidocs/config/config.md`).
  - Overview/reference hubs link to generated module docs under `apidocs/`.
  - Reviewer navigation: spot‑check `apidocs/core/core.execution.launcher.md` and `apidocs/cli/cli.md` for coverage.

- [ ] Integrations and prerequisites
  - Monitoring tools are framed as usage patterns in `integrations/monitoring-tools.md` (not bundled plugins).
  - Ray content clearly calls out KubeRay custom resources (`RayCluster`, `RayJob`) in `guides/ray.md`; include links to CRDs/setup where available.
